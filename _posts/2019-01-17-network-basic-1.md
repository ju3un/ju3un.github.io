일단 네트워크 프로그래밍에서 가장 기본적인 개념에 대해 정리할 예정이다.<br>

#Blocking vs Non-Blocking

##1. I/O 작업

먼저 I/O 용어는 Input/Output의 약자로 알고 있을 것이다. 주로 파일 입출력을 다룰 때 흔히 볼 수 있다.<br>
네트워크에서도 쉽게 볼 수 있는데 예를 들어, 소켓의 read/send를 생각하면 이해가 갈 것이다.

좀 더 쉽게 예를 들어보겠다.<br>
두 대 이상의 컴퓨터끼리 서로 네트워크를 통해 통신을 한다고 가정할 때,<br>
한 컴퓨터에서 출력(send)을 하고, 다른 한 컴퓨터에서 입력(read)을 받는 과정을 통해 통신을 할 수 있다.

I/O 작업은 User(유저 레벨)에서 직접 수행할 수 없고, **실제 IO 작업을 수행하는 것은 Kernel(커널 레벨)**에서만 가능하다.<br>
**유저 프로세스(또는 스레드)는 커널에게 요청을 하고 작업 완료 후 커널이 반환하는 결과를 기다릴 뿐이다.**

그렇다면 이제 시작해보자.

---

##2. Blocking Model

가장 기본적인 I/O 모델로, linux에서 모든 소켓 통신은 기본 blocking으로 동작한다.<br>
**I/O 작업이 진행되는 동안 유저 프로세스는 자신의 작업을 중단한 채 대기하는 방식이다.**

![](https://user-images.githubusercontent.com/41428527/51262623-1070fc00-19f6-11e9-9ec5-131c90c01ecb.png)

위의 그림처럼 
>1. 유저는 커널에게(유저->커널) read작업을 요청하고 
>2. 데이터가 입력될 때까지 대기하다가 
>3. 데이터가 입력되면 유저에게(커널->유저) 결과가 전달되어야만 유저 자신의 작업에 비로소 복귀할 수 있다.

**말 그대로 block이 되고, 어플리케이션에서 다른 작업을 수행하지 못하고 대기하게 되므로 자원이 낭비된다.**<br>

>진짜 다른 예를 들어보면, 카카오톡을 생각해보자.<br>
카카오톡이 사용자가 메세지를 전송할 때까지 대기하고 있는거라고 생각하면 이해가 쉬울 것 같다.

>**"너 보낸 메세지 있어?" -> .......(무한 대기 중) '도대체 언제 보낼꺼야...'**

>요즘 카카오톡은 메신저 이외에도 다양한 기능이 많은데<br>
그저 하염없이 우리 메세지만 기다리고 있는 아이라고 생각하면 얼마나 안타까운가...<br>

***

##3. Non-Blocking Model

위와 같은 blocking 방식의 비효율성을 극복하고자 도입된 방식이다.<br> 
**I/O 작업이 진행되는 동안 유저 프로세스의 작업을 중단시키지 않는 방식이다.**
>**어떻게?**

![](https://user-images.githubusercontent.com/41428527/51262628-136bec80-19f6-11e9-890c-0caf3c4c2a2b.png)

>1. 유저가 커널에게 (유저->커널) read작업을 요청하면
>2. **데이터가 입력이 됬든 안됬든 요청하는 그 순간, 바로 결과가 반환된다.**<br>
   이 때, 입력 데이터가 없으면 입력 데이터가 없다는 결과 메세지(EWOULDBLOCK)를 반환한다.
>3. 입력 데이터가 있을 때 까지 **1-2번 반복.** (**2번에서 결과 메세지를 받은 유저는 다른 작업 진행이 가능하다.**) 
>4. 입력 데이터가 있으면 유저에게(커널->유저) 결과가 전달된다.

**이 경우 I/O의 진행시간과 관계가 없기 때문에(대기x) 어플리케이션에서 작업을 오랜 시간 중지하지 않고도 I/O 작업을 진행할 수 있다.**<br> 
**그러나 반복적으로 시스템 호출이 발생하기 때문에 이 경우 역시 자원이 낭비된다.**

>이 방식의 카카오톡은 내가 메세지를 보낼 때 까지 바보같이 대기하지 않을 것이다.

>**"너 보낸 메세지 있어?" -> "없구나!" -> '잠시 이 사용자한테 공지좀 보내야지!' -> "너 보낸 메세지 있어?"(반복)**

>이렇게 카카오톡은 우리가 알던 그 바보가 아니다! **하지만 계속 했던 말 계속 반복하는 건.. 역시 바보가 맞다....(자원 낭비!)**

>역시 Non-Blocking의 비효율성을 극복하기 위해 **I/O 이벤트 통지 모델**이 도입되었다.<br>
